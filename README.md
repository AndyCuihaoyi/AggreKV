# AggreKV

[![DOI](https://zenodo.org/badge/DOI/10.5281/zenodo.21502922.svg)](https://doi.org/10.5281/zenodo.21502922)

Source tree for the paper *AggreKV: Boosting Hash-based KV-SSD Performance via Locality-Centric Optimizations*.

This repository contains a user-space, kernel-bypass KV-SSD simulator written in C and ships three independent implementations:

- `hash_kvssd/` — AggreKV, the proposed hash-based KV-SSD with locality-centric optimizations (`HOT_CMT`, `ADAPTIVE_MEM`, `DATA_SEGREGATION`).
- `lsmtree_kvssd/` — LSM-tree baseline (WiscKey-style).
- `block_ssd/` — Block-SSD baseline (DFTL-style). Source included; see `INSTALL.md` §3 for how to build it and §3.1 for the dataset fetch.

## 3. What This Project Is

This repository is a **user-space, kernel-bypass KV-SSD simulator** written in C. The simulator emulates SSD behaviour in DRAM; no real NVMe device is required. It ships three independent implementations:

| Sub-module       | Role in the paper |
|------------------|-------------------|
| `hash_kvssd`     | AggreKV — the proposed hash-based KV-SSD with locality-centric optimizations (`HOT_CMT`, `ADAPTIVE_MEM`, `DATA_SEGREGATION`). |
| `lsmtree_kvssd`  | LSM-tree baseline (WiscKey-style). |
| `block_ssd`      | Block-SSD baseline (DFTL-style) driven by RocksDB-derived blktrace traces. |

## 4. Hardware & Software Requirements

- Linux x86_64 (Ubuntu 22.04 LTS recommended).
- `gcc 11.x`, GNU `make`, `bash 5.x`.
- `libglib2.0-dev` (required by the `hash_kvssd` baseline; `block_ssd` and `lsmtree_kvssd` only link against `-lglib-2.0`, so on Debian/Ubuntu they don't strictly need the dev headers).
- `numactl` (optional; auto-fallback to `taskset -c 0`).
- 256 GB RAM, single NUMA node binding; ~100 GB free disk at the clone location.
- The 6.2 GB blktrace dataset, fetched separately per `INSTALL.md` §3.1.

The full checklist (with `Disallowed Modifications`, run-output notes, and tested-platform rows) is in `REQUIREMENTS.md`.

## 5. Quick Start

From a fresh Linux x86_64 box:

```bash
# 1. OS-level build dependencies
sudo apt update
sudo apt install -y build-essential libglib2.0-dev numactl

# 2. Obtain the source
git clone https://github.com/AndyCuihaoyi/AggreKV.git AggreKV
cd AggreKV

# 3. Fetch the blktrace dataset (RocksDB-derived, ~6.2 GB total).
#    See INSTALL.md §3.1 for details; the script downloads 44 files in
#    4 parallel wget jobs and verifies each file's SHA256.
bash download-blktrace.sh

# 4. Build the block_ssd sub-module.
#    hash_kvssd and lsmtree_kvssd build themselves on first invocation, so
#    no pre-build step is needed for them.
( cd block_ssd && make test_blktrace test_ext_mem_lat )

# 5. Run the orchestrator
bash test.sh
```

Hardware / NUMA / RAM constraints and the full troubleshooting table are in `INSTALL.md` and `REQUIREMENTS.md`.

## 6. Outputs and How to Interpret Them

Each sub-driver writes to its own results directory:

| Sub-module       | Output directory                              | Key summary file                                |
|------------------|-----------------------------------------------|--------------------------------------------------|
| `hash_kvssd`     | `hash_kvssd/hash_kvssd_results/`              | `summary/summary.txt`, `summary/summary.csv`     |
| `lsmtree_kvssd`  | `lsmtree_kvssd/lsmtree_test_results/`         | `summary.txt`, per-workload `*.log` files |
| `block_ssd`      | `block_ssd/blktrace_test_results/` and `block_ssd/blktrace_summary/` | `summary/summary_<timestamp>.txt` |

A successful run means each script ran to completion and produced well-formed output under its own results directory.
