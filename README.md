## 4. What This Artifact Is

A **user-space, kernel-bypass KV-SSD simulator** written in C. The simulator emulates SSD behaviour in DRAM; no real NVMe device is required. The artifact ships three independent implementations:

| Sub-module       | Role in the paper |
|------------------|-------------------|
| `hash_kvssd`     | AggreKV — the proposed hash-based KV-SSD with locality-centric optimizations (`HOT_CMT`, `ADAPTIVE_MEM`, `DATA_SEGREGATION`). |
| `lsmtree_kvssd`  | LSM-tree baseline (WiscKey-style). |
| `block_ssd`      | Block-SSD baseline (DFTL-style) driven by RocksDB-derived blktrace traces. |

`test.sh` is a thin orchestrator: it `cd`s into each sub-module and runs that sub-module's own driver in turn, then exits 0 if all three succeed.

## 5. Hardware & Software Requirements

- Linux x86_64 (Ubuntu 22.04 LTS recommended).
- `gcc 11.x`, GNU `make`, `bash 5.x`.
- `libglib2.0-dev` (required by the `hash_kvssd` baseline; `block_ssd` and `lsmtree_kvssd` only link against `-lglib-2.0`, so on Debian/Ubuntu they don't strictly need the dev headers).
- `numactl` (optional; auto-fallback to `taskset -c 0`).
- 256 GB RAM, single NUMA node binding; ~100 GB free disk at the clone location.
- The 6.2 GB blktrace dataset, fetched separately per `INSTALL.md` §3.1 (only needed for `block_ssd`).

The full checklist (with `Disallowed Modifications`, run-output notes, and tested-platform rows) is in `REQUIREMENTS.md`.

## 6. Quick Start

First, follow `INSTALL.md` to install OS dependencies, fetch the blktrace dataset (if you want the Block-SSD baseline to actually run), and build the `block_ssd` sub-module.

Then, from the repository root:

```bash
git clone <TODO: GitHub repo URL> aggrekv-artifact
cd aggrekv-artifact
bash test.sh
```

`test.sh` prints

```
[TEST OK] all three sub-drivers finished
```

and exits 0. Detailed step-by-step install instructions (apt packages, the block_ssd build step, and the blktrace download step) live in `INSTALL.md`.

## 7. What `test.sh` Runs

| Step | Sub-module       | Driver invoked                                            |
|------|------------------|-----------------------------------------------------------|
| 1    | `hash_kvssd`     | `bash run_all_hashkvssd_tests.sh` (sub-driver builds internally) |
| 2    | `lsmtree_kvssd`  | `bash run_lsmtree_tests.sh` (after `make lsm_test`)       |
| 3    | `block_ssd`      | `bash run_blktrace_tests.sh` (after `make test_blktrace`) |

The three sub-drivers are independent and can be built and run in any order. The top-level `test.sh` is a thin orchestrator; it does **not** aggregate results, does **not** claim numerical alignment with the paper, and does **not** define a `MODE` flag.

If `block_ssd/blktrace/` is empty or missing (the standard state on a fresh GitHub clone), `run_blktrace_tests.sh` exits 0 with a clear skip message. See `INSTALL.md` §3.1 to fetch the dataset if you want the Block-SSD baseline to actually run.

To exercise a single sub-driver or to pass custom arguments (e.g. `--smoke`, `--only E1`, `--num N`), invoke that sub-driver directly — see the header of each sub-driver script for its flag catalog.

## 8. Outputs and How to Interpret Them

Each sub-driver writes to its own results directory:

| Sub-module       | Output directory                              | Key summary file                                |
|------------------|-----------------------------------------------|--------------------------------------------------|
| `hash_kvssd`     | `hash_kvssd/hash_kvssd_results/`              | `summary/summary.txt`, `summary/summary.csv`     |
| `lsmtree_kvssd`  | `lsmtree_kvssd/lsmtree_test_results/`         | `summary.txt`, per-workload `*.log` files, `test_config.txt` |
| `block_ssd`      | `block_ssd/blktrace_test_results/` and `block_ssd/blktrace_summary/` | `summary/summary_<timestamp>.txt` |

These directories are `.gitignore`d; they are regenerated by each sub-driver and overwritten in place on re-run.

To confirm that the Block-SSD baseline **actually consumed the blktrace dataset** (rather than skipping for lack of it), check that, after `bash test.sh` finishes, `block_ssd/blktrace_test_results/` contains one `test_*.log` per workload (i.e. one per `.csv` in `block_ssd/blktrace/`) and that `block_ssd/blktrace_summary/summary_<timestamp>.txt` is non-empty. If both are true, the download + replay succeeded; if `blktrace_test_results/` is empty or missing, see `INSTALL.md` §3.1 — most often the dataset was not downloaded, and `run_blktrace_tests.sh` skipped that baseline.

A successful run demonstrates that the artifact is documented, consistent, complete, exercisable, and produces well-formed output. It does **not** guarantee that the numerical results match the figures in the paper. Numerical reproduction of the paper's figures is the separate Reproduced badge, which we are not applying for.
