## 5. Quick Run — `bash test.sh`

The top-level `test.sh` is a thin entry point: it `cd`s into each of the three sub-modules and runs that sub-module's own driver in order, then exits 0 if all three finished successfully. There is **no `MODE` flag at the root level**, no aggregated `results/` directory, and no `STATUS.txt` / `SUMMARY.md` — each sub-driver writes its own outputs into its own results directory (see §7).

```bash
# from the repository root
bash test.sh
```

**Success marker:**

```
[TEST OK] all three sub-drivers finished
```

If you see `[TEST OK]`, proceed to §7 (Sanity-Check Outputs). If `test.sh` exits non-zero, the failing sub-driver's log is the first place to look; consult [INSTALL.md](INSTALL.md) §6 for common build errors.

**What `test.sh` runs (in order):**

| Step | Sub-module       | Driver invoked                                                  |
|------|------------------|-----------------------------------------------------------------|
| 1    | `hash_kvssd`     | `bash run_all_hashkvssd_tests.sh`                               |
| 2    | `lsmtree_kvssd`  | `bash run_lsmtree_tests.sh` (after `make lsm_test` inside)     |
| 3    | `block_ssd`      | `bash run_blktrace_tests.sh` (after `make test_blktrace` inside) |

The three sub-drivers are completely independent: they share no build state, can be built and run in any order, and each writes only into its own sub-module directory.

> **Smoke vs. full data volumes.** The root-level `test.sh` always runs each sub-driver with its own defaults. To exercise the smoke flag, custom KV counts, or a single experiment, run that sub-driver directly — for example:
>
> ```bash
> cd hash_kvssd    && bash run_all_hashkvssd_tests.sh --smoke
> cd lsmtree_kvssd && bash run_lsmtree_tests.sh --num 100000
> cd block_ssd     && bash run_blktrace_tests.sh
> ```
>
> See §8 for the full script catalog.
